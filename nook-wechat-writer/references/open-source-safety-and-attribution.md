# 1 开源安全与归因原则

这份 reference 约束 nook 系列 skills 的写作、整理和发布。

nook skills 未来计划开源，所以从草稿阶段开始，就按可发布标准处理归因、隐私、敏感信息和外部参考。

## 1.1 发布定位

nook 是一套面向个人创作者的内容生产工作流。

它可以受到优秀创作者和既有内容系统的启发，但最终发布时必须体现自己的方法论，而不是某个创作者的复刻包。

正式开源版本要让用户看到：

- 这是一套独立的内容生产框架。
- 它适合迁移到不同创作者身上。
- 它不依赖某个私人 vault、账号、数据库或内部系统。
- 它没有泄露作者本人或他人的敏感信息。

## 1.2 归因原则

可以轻量说明 nook 的灵感来源。

不要把灵感来源写成核心卖点。

推荐写法：

> nook 的内容生产方法受到多个中文 AI 内容创作者的启发，尤其是“流程化生产”和“真实表达”这两类方法意识。nook 不是对任何个人风格的复刻，而是面向个人创作者的独立内容生产工作流。

不推荐写法：

> 本 skill 复刻某某创作者。

> 本 skill 是某某风格生成器。

> 本 skill 完整搬运某某创作系统。

正式 skill 的名称、description、默认角色设定和触发词中，不要把外部创作者作为核心身份。

## 1.3 蒸馏而不是拼贴

使用外部参考时，只吸收抽象原则，不复制强识别表达。

可以吸收：

- 选题要先判断，不要直接开写。
- 新主题要先做信息挖掘。
- 内容要保留真实观察、明确判断和具体细节。
- 审校要分层，而不是一次性随机重写。
- 视频逐字稿要优先保证听感。

不要复制：

- 外部创作者的完整段落。
- 外部创作者的专属口头禅。
- 外部创作者的强识别案例组合。
- 外部创作者的人设自述。
- 外部创作者的原始文件结构和私有工作流命名。

如果一个规则离开原作者仍然成立，就抽象成 nook 规则。

如果一个规则只有模仿原作者时才成立，就不要写进 nook。

## 1.4 敏感信息禁区

正式开源内容中不得出现：

- API key、access token、refresh token、app secret。
- Cookie、Authorization header、session 信息。
- 数据库 ID、表 ID、视图 ID、文档 token、工作区 ID。
- 私有 Webhook、服务器地址、内网地址、云函数地址。
- 真实邮箱、手机号、身份证号、住址、公司内部通讯录。
- 未公开客户名称、商业项目、合同信息、报价信息。
- 本地绝对路径，例如 Windows 盘符路径、用户主目录路径。
- `.env` 文件内容。
- 私有账号名称、机器人名称、组织名称。
- 可定位到个人或客户的真实素材。

本地设计笔记可以临时包含私人信息，但不得进入正式开源目录。

## 1.5 占位符规范

需要配置时，使用占位符：

- `<YOUR_API_KEY>`
- `<YOUR_ACCESS_TOKEN>`
- `<YOUR_DATABASE_ID>`
- `<YOUR_TABLE_ID>`
- `<YOUR_VAULT_PATH>`
- `<YOUR_ACCOUNT_NAME>`
- `<YOUR_WEBHOOK_URL>`

需要环境变量时，只提供 `.env.example`，不要提供 `.env`。

示例要使用假值：

```env
OPENAI_API_KEY=<YOUR_API_KEY>
DATABASE_ID=<YOUR_DATABASE_ID>
```

不要使用看起来像真实 key 的假值。假的 key 也会误导用户。

## 1.6 本地版与开源版分层

建议保留两层目录：

- 设计工作区，保存思考、草稿、私人测试和本地路径。
- 发布工作区，只保存可开源版本。

从设计工作区复制到发布工作区前，必须做脱敏检查。

不要直接把本地设计目录发布到 GitHub。

## 1.7 发布前安全检查

发布前搜索这些关键词：

- `sk-`
- `api_key`
- `apikey`
- `token`
- `secret`
- `password`
- `cookie`
- `authorization`
- `bearer`
- `app_id`
- `app_secret`
- `database_id`
- `table_id`
- `base_id`
- `view_id`
- `webhook`
- `.env`
- `C:\`
- `D:\`
- `I:\`
- `Users\`
- `邮箱`
- `手机号`
- `身份证`

命中后逐项判断。

真实敏感信息必须删除或替换为占位符。

私人路径必须改成相对路径或 `<YOUR_VAULT_PATH>`。

内部项目名必须泛化。

## 1.8 发布前归因检查

发布前检查这些问题：

- 有没有把外部创作者写成 skill 的核心角色？
- 有没有直接复述外部创作者的大段原文？
- 有没有保留外部创作者强识别的口头禅、案例和人设？
- 有没有把“参考”写成“复刻”？
- 用户能不能在不知道外部创作者是谁的情况下，仍然理解和使用 nook？

如果答案不理想，就继续抽象。

## 1.9 合格标准

一份可开源的 nook reference 应该满足：

- 不含真实密钥、token、数据库 ID 和私人路径。
- 不含未公开商业信息和可识别个人隐私。
- 不依赖作者本人的 vault 结构。
- 有清晰的 nook 自己的方法论。
- 可以致谢灵感来源，但不把灵感来源当作产品主体。

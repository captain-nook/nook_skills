# Nook Skills

这是 Nook 的公开 Codex Skills 发布库。
放已经沉淀成可复用能力的 skill：写作、审校、标题、视觉生成、封面生产、图片 provider，以及少量第三方能力封装。
目标是让以后新增 skill 时，能持续沉淀三件事：

- 这个 skill 当初为了解决什么问题而做；
- 遇到某类工作时应该调用哪个 skill；
- skill 之间如何分工，避免重复造轮子。

## 1. 快速选择

| 你要做的事 | 优先调用 | 说明 |
|---|---|---|
| 写微信公众号长文、改稿、从素材成文 | `nook-wechat-writer` | 负责文章角度、结构、正文、摘要、跨平台改写。 |
| 给已经定稿的文章取标题 | `nook-title` | 负责标题事实钩子、候选标题、标题评分和事实边界。 |
| 判断一段中文有没有 AI 味、做自然化审校 | `nook-humanizer-zh-review` | 作为写作 skill 的最终审校层，也可单独使用。 |

| 做小红书/公众号/B站等封面，尤其有标题文字层 | `nook-cover` | 多平台封面生产主 skill，当前重点覆盖小红书封面。 |

## 2. 内容生产类

### `nook-wechat-writer`

公众号文章生产主 skill。

它的核心价值是把“想法、素材、转录稿、链接、粗稿”变成一篇可发布的公众号文章。它不是简单写作提示词，而是一套流程：先确认价值点和角度，再组织结构、写正文、做事实检查、做风格审校。

适用场景：

- 从一堆素材里写出公众号文章；
- 修改已有文章，让结构更清楚；
- 把视频脚本、会议记录、读书笔记改成文章；
- 做文章角度确认、摘要、发布字段；
- 文章与短视频脚本之间互相改写。

不要用它做：

- 单独生成标题：用 `nook-title`；
- 单独去 AI 味：用 `nook-humanizer-zh-review`；
- 发布排版、头图尾图装配：应由专门的发布/packager skill 负责。

### `nook-title`

标题生成和标题评分 skill。

它解决的是一个很具体的问题：稿件已经写完了，接下来要取几个标题，但标题既要有点击理由，又不能编数字、编榜单、把传言写成事实。

适用场景：

- 在 Obsidian 中 `@` 一篇定稿，然后说“给我取几个标题”；
- 给 AI 资讯、公众号文章、产品更新、工具评测生成候选标题；
- 评估已有标题哪个更适合发布；
- 从一批标题中提炼长度、标点、数字、实体名等规律。

它会先抽事实钩子：实体、数字、动作、冲突、后果、人群、事实边界。再生成高点击版、稳一点的公众号版、搜索友好版，并标注风险。

### `nook-humanizer-zh-review`

中文去 AI 味审校 skill。

它不是“把文字改口语一点”，而是检查文本是不是有典型 AI 痕迹：空泛转折、整齐排比、过度总结、假装有观点、缺少具体细节。

适用场景：

- 文章最终润色前做一遍人味审校；
- 检查视频脚本是否像人在说话；
- 把过于顺滑、过于报告腔的段落改自然；
- 作为 `nook-wechat-writer` 的强制子审校层。

## 3. 视觉与封面类

### `nook-card`

卡片设计 skill。

它用于把信息做成有结构、有视觉语言的卡片，而不是简单把文字贴到图上。适合知识卡片、方法卡片、信息图、社媒传播图。

沉淀原因：

- 卡片类输出容易变成模板堆叠；
- 需要把内容结构、视觉层级、交互动效和交付规格稳定下来；
- 后续可以继续接入 OpenDesign、Remotion、HTML 模板等工作流。

### `nook-cover`

多平台封面生产主 skill。

它面向更完整的封面工作流：理解平台、确定主视觉、生成背景或使用素材、稳定中文标题层、导出对应尺寸。

适用场景：

- 小红书封面；
- 公众号封面；
- B站/YouTube/抖音/视频号等平台封面规划；
- 需要标题文字层稳定落版的封面。

### `nook-tv-cover`

视频封面专项 skill。

它聚焦横封面、短封面、竖封面三类视频平台封面，默认偏科技风、内容感、信息密度较高的视觉方向。

适用场景：

- YouTube 长封面；
- B站短封面；
- 抖音/小红书/视频号竖封面；
- 视频内容主视觉与标题组合。

## 4. 图片 Provider 类

这些 skill 更像底层能力，不一定直接面向最终内容。上层视觉 skill 可以根据质量、成本、速度来调用它们。

### `nook-image-gpt`

早期 GPT Image 相关图片生成 provider。保留用于兼容已有流程。

### `nook-image2-gpt`

更偏原子能力的 GPT Image provider，适合稳定的文生图和图生图任务，也适合被更大的视觉 skill 调用。

### `nook-zimage`

低成本快速文生图 provider。

适合：

- 草稿探索；
- 批量试方向；
- 背景图、低风险素材；
- 不需要最高质量但需要速度和成本控制的任务。

### `nook-qwen-image`

中文视觉质量更高的图片 provider。

适合：

- 中文海报；
- 中文封面；
- 标题型视觉；
- 对中文文字渲染要求更高的图像任务。

## 5. 第三方能力

### `third_party/Humanizer-zh`

第三方中文去 AI 检测/改写能力保留区。

它不是 Nook 原生写作方法本身，而是作为参考和对照，帮助 `nook-humanizer-zh-review` 沉淀更具体的审校规则。

## 6. 维护原则

### 6.1 什么应该进入这个仓库

- 已经可以被 Codex 调用的发布态 skill；
- 对应的 `SKILL.md`、`agents/openai.yaml`、`references/`、`scripts/`、必要资产；
- 能解释 skill 使用边界的 README 或参考文件；
- 可复用、可迁移、可公开的工作流。

### 6.2 什么不应该进入这个仓库

- Obsidian 笔记库内容；
- 个人素材、临时讨论、未清洗截图；
- 私密路径、token、账号信息；
- 只服务某一次任务的过程文件；
- 还在建设中的半成品 skill。

建设态内容应该先放在私有工作区，确认可复用后再发布到这里。

### 6.3 新增 skill 时要补什么

新增一个 skill 时，至少补齐：

- `SKILL.md`：触发描述要清楚，说明什么时候用；
- `README.md`：给人看的用途说明；
- `agents/openai.yaml`：给 Codex UI 识别；
- `references/`：只放真正需要复用的规则和样例；
- `scripts/`：只放确定性高、反复会用的脚本；
- 根 README：把这个 skill 加入“快速选择”和对应分类。

# 1 Component Contract

This is the planning-facing component contract for the current release.

The Agent may choose approved components and variants. The Agent may fill named fields. The Agent must not create visual layout.

The 13 approved components are global. They are not owned by one visual system.

Every production visual system must implement the same component IDs, natural page forms, variants, field slots, capacity rules, and relative structural skeletons. A visual system may only change the skin: colors, fonts, stroke style, corner radius, texture, shadows, image treatment, and decorative motifs. It must not add, remove, rename, or rearrange component slots.

The machine-readable skeleton file for visual-system component libraries is:

- `assets/templates/global-component-skeleton-v1.json`

That file is a generated artifact, not a hand-authored design source. Generate it from:

- source: `assets/templates/duotone-zine/component-library-source.py`
- extractor: `assets/templates/extract_duotone_component_skeleton.py`

Theme sample libraries must be generated by applying theme tokens to the extracted skeleton through `assets/templates/generate_theme_component_libraries.py`. If a component library is drawn from a theme-specific layout script, or if `global-component-skeleton-v1.json` is edited by hand instead of extracted from the duotone source, treat it as invalid for the current contract.

## 1.2 Approved Components

| Component | Use when | Main slots | Capacity rule |
|---|---|---|---|
| `zine.hero-title` | cover, opener, case opener | `title`, `subtitle`, `body`, optional `image_slots`, optional `chips` | max 1 image |
| `zine.statement` | one strong judgment or transition | `title`, `body`, optional `chips`, optional `cards` | 0-3 chips; fragment variant 2-4 cards |
| `zine.big-number` | three metrics or three punchy actions | `title`, `body`, `number`, `cards` | exactly 3 cards |
| `zine.process-cards` | workflow, method, chain, deliverables | `title`, `body`, `cards`, `footer` | 3-5 cards by variant |
| `zine.timeline` | chronological order, layers, CPR-like sequence | `title`, `body`, `cards` | 3, 4, or 5 cards by variant |
| `zine.cycle` | closed loop or repeated rhythm | `title`, `body`, `cards`, `footer` | exactly 4 cards |
| `zine.hierarchy` | layered system, stack, dependency | `title`, `body`, `cards`, `footer` | 3-6 cards by variant |
| `zine.comparison` | before/after, A/B, old/new, option comparison | `title`, `body`, `cards`, `footer` | 2-5 cards by variant |
| `zine.card-group` | three peer ideas with no direction or hierarchy | `title`, `body`, `cards` | exactly 3 cards |
| `zine.image-text` | one dominant image/screenshot plus short explanation | `title`, `body`, `footer`, `image_slots`, optional `chips` | exactly 1 image |
| `zine.image-collage` | three images/screenshots as evidence | `title`, `body`, `image_slots` | exactly 3 images |
| `zine.brand-master` | logo/IP/brand asset rules | `title`, `body`, `cards`, `image_slots` | 2-4 cards; 1-4 images |
| `zine.generated-asset` | illustration or generated-asset specification | `title`, `body`, `cards`, optional `image_slots` | 3-4 cards; max 1 image |

## 1.3 Planning Names

When talking to the user, use natural Chinese page forms:

- 封面页
- 一句话判断页
- 大数字页
- 流程页
- 时间轴页
- 循环页
- 层级页
- 对比页
- 并列卡片页
- 图文页
- 图片拼贴页
- 品牌资产页
- 出图资产页

Do not ask the user to approve `zine.*` IDs, variants, slot names, or validation fields. Translate user-approved page forms into `zine.*` IDs internally.

## 1.4 Variant Discipline

Variants are approved capacities, not style suggestions.

- If content has more items than the selected variant supports, reopen the page-form gate.
- Split the page or choose another approved variant.
- Do not shrink type, trim confirmed content, or create a generic card grid.

## 1.5 Text Discipline

Every visible field must be short enough to fit its component. The schema contains maximum lengths. If the source has too much text, summarize before the content gate and ask the user to approve the shorter visible copy.

After approval, do not rewrite visible copy during schema filling or rendering.

## 1.6 Visual Carrier

Every ordinary page needs a declared `visual_carrier`:

- `statement`
- `image`
- `screenshot`
- `chart`
- `table`
- `structure`

Use `statement` only for approved thinking-prompt or judgment pages. Prefer real screenshots/images for factual claims.
